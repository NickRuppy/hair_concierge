AgentV2 guidance migration

Where should request_interpretation sit?

This map explains the decision in plain architecture terms: which layer should reason broadly, which layer should declare the final answer contract, and which layer should enforce product and safety truth.

The pipeline we are deciding around

1. User message Natural German request. Often messy: symptoms, product asks, routine follow-ups, or safety signals.
2. Model reasoning GPT-5.4-mini interprets meaning, ambiguity, tone, and which context/tools might help.
3. load_advisor_guidance Loads one or more guidance packages: base, category, routine, safety, and general haircare.
4. Tools and catalog Product facts, IDs, availability, ordering, protocol, and routine state come from tools, not guidance prose.
5. submit_final_answer Final visible answer plus request_interpretation, which the validator checks.

Recommended decision

Keep request_interpretation narrow. It should be the final accountability contract, not the broad reasoning map.

  • care_category names the central product or care category only when one is central.
  • load_advisor_guidance may load multiple packages for reasoning.
  • Validators use the interpretation to check tool use, grounding, answer mode, product count, safety, and routine permissions.

The trap to avoid

If request_interpretation becomes "all topics the model considered," it turns into a hidden router taxonomy. That makes it bigger, harder to validate, and less useful as a final contract.

  • Concerns like dryness and frizz can inform reasoning.
  • They do not always need to become terminal schema fields.
  • Broad reasoning belongs in guidance loading, answer synthesis, traces, and evals.

Three options

A. Narrow final contract

Recommended

request_interpretation describes the answer the agent actually gave. Guidance loading can be broad and multi-package.

Best when You want strong validation without rebuilding a router.
Pros Small schema, easy evals, clear grounding rules, natural model reasoning.
Cons Traces need to show loaded guidance if you want to inspect all reasoning context.

B. Broad topic contract

Tempting

request_interpretation lists every relevant topic: category, concern, routine layer, technique, safety, and profile signals.

Best when You need exhaustive analytics more than lean validation.
Pros Rich logs, easier topic coverage analysis, explicit multi-topic visibility.
Cons Schema grows fast, validators become brittle, and the field starts acting like a router.

C. No request interpretation

Too loose

The model answers and tools run, but there is no terminal semantic declaration for code to validate against.

Best when You only need a casual chat assistant with low consequence.
Pros Less schema work and fewer repair loops.
Cons Harder to catch unasked products, wrong counts, skipped tools, unsafe escalation, and routine mutation mistakes.

What each layer should own

Layer Owns Should not own
load_advisor_guidance Broad context selection: base rules, product categories, safety, routine guidance, and general haircare. Final semantic truth, product IDs, product availability, or saved routine mutations.
request_interpretation The final answer contract: intent, product request kind, routine intent, central care_category, count, evidence quote, and confidence. Every topic used for reasoning, every concern mentioned, or a complete retrieval trace.
Product and routine tools Facts: product IDs, ranking, catalog membership, lifecycle, protocols, routine state, and persisted changes. Free-form category theory or hidden product claims from prose guidance.
Validator Consistency: did the answer match the declared interpretation, loaded guidance, required tools, product count, and safety boundary? Natural-language taste, empathy, or nuanced explanation style unless it violates a hard rule.
Plain-English verdict

Let GPT-5.4 reason broadly. Let load_advisor_guidance bring in the useful context. Let request_interpretation stay small enough that code can hold the answer accountable. That is the clean split between model intelligence and deterministic product safety.
