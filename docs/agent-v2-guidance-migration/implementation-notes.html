AgentV2 Implementation Notes

Running notes for the AgentV2 guidance rehydration and contract-fix implementation. This file captures implementation decisions that interpret, narrow, or intentionally depart from the plan.

Design Decisions

2026-05-19: Continue in the existing feature worktree codex/gpt-54-responses-migration-plan.

The worktree is dirty, but the changed files are already in the AgentV2 and plan area for this task. I am not creating a second worktree because that would make it harder to preserve the Compare Lab state and prior AgentV2 edits.

2026-05-19: The controller owns this notes file.

Subagents should report decisions, deviations, tradeoffs, and open questions, but should not edit implementation-notes.html. This avoids write conflicts while keeping a readable implementation log for review.

2026-05-19: Restricted safety repair stays terminal-first.

If a rejected product-shaped answer fails because of safety_no_product_first, the repair path does not request select_products even when product_tool_required is also present. Restricted safety intentionally removes product selection from the safe tool surface, so the correct repair is a safety answer, not a product lookup.

2026-05-19: Required guidance is now checked against runtime-loaded guidance when a normal terminal answer is validated.

The terminal contract still carries tool_grounding.used_guidance_package_ids, but the runtime passes trace.loaded_guidance_package_ids into the validator so the model cannot satisfy task/category guidance requirements through self-reporting alone.

2026-05-19: Stable runtime instructions count as loaded base contract guidance.

The runtime now seeds base.advisor_rules.v1, base.answer_contract.v1, and base.tone_and_format.v1 into the trace because those constraints are present in stable system instructions on every turn. Mode-specific and category-specific packages still need actual load_advisor_guidance output.

2026-05-19: Ordered repair tools are enforced before accepting another terminal answer.

If validation says both guidance and product grounding are missing, the repair path now requires load_advisor_guidance before select_products, then submit_final_answer. A premature terminal answer during that repair is blocked and falls back safely instead of silently passing.

2026-05-19: Safety guidance loads from answer-mode intent as well as safety mode.

load_advisor_guidance now includes base.safety_boundaries.v1 when answer_mode_hint is safety_boundary, even if the external safety mode is still normal. This covers cases where the model recognizes the boundary inside the loop.

2026-05-19: Bondbuilder pilot keeps old confusion guidance as model-facing category context, not regex-style validation.

The old bond-builder/confusions.md content was converted into compact category.bondbuilder.v1 sections plus traceable hard rules and a soft rubric. I intentionally removed the placeholder category_claim_boundary validator ID because that validator does not exist and would imply deterministic claim scanning we have not built.

2026-05-19 correction: The Bondbuilder pilot now treats the full old topic folder as the migration unit.

The first pilot was too lossy because it centered confusions.md. The corrected package now incorporates core-fit.md, response-playbook.md, guardrails.md, confusions.md, and the README load-order context into one AgentV2 category package.

2026-05-19: Broader old guidance was folded into existing category and base packages.

Following the Bondbuilder pilot, the category/base migration workers enriched the existing AgentV2 packages instead of creating a runtime overlay system. The old overlays are represented as compact best-fit, weak-fit, confusion, grounding, and answer-shape guidance in the relevant packages.

2026-05-19: The source map is an inventory and audit aid, not a claim that every sentence was copied.

The map now marks old guidance files as considered for V0. That wording is deliberate: the migration goal is to preserve usable logic in GPT-5.4-friendly packages, while leaving obsolete, duplicate, or over-specific wording behind.

2026-05-19: The broad prompt runner is a manual-review report, not an automated quality verdict.

scripts/agent-v2/run-guidance-regression.ts runs the agreed prompt batch against AgentV2 and writes JSON/Markdown reports under tmp/. It marks deterministic misses as fail and normal answer-quality cases as review, so we can inspect actual final answers without pretending string heuristics are a judge.

Deviations

None currently.

Tradeoffs

Subagent execution order: I am starting with the runtime/validator contract work before broad guidance migration.

This keeps the later Bondbuilder and category guidance tests meaningful: if the terminal contract or repair path is still broken, guidance quality would be hard to judge.

Subagent scope: The first worker owns only Phase 1-2 runtime, validator, tool definition, eval, and directly related test changes.

Broad guidance content migration is intentionally held back until the core contract and repair behavior are stable.

Required guidance fixture cleanup: Older passing validator fixtures were updated to include derived required guidance IDs.

This is fixture alignment rather than a new product behavior: the validator now enforces always-base guidance plus answer-mode and category guidance for non-trivial answers.

Repair prioritization: Answer-mode mismatches repair terminal-first instead of loading guidance for the wrong mode.

When the model declares a concrete product intent but submits general_advice, the validator may also notice that general-advice guidance is missing. In that specific combination, the repair should correct the terminal answer to product mode using already returned product evidence, not load unrelated general guidance.

Bondbuilder examples: High-confidence technologies from the old doc were kept as explanatory context, not default recommendations.

This preserves the useful distinction while keeping product selection authoritative: concrete Bondbuilder product names, availability, order, and category membership still need select_products or curated catalog data.

Bondbuilder product facts: Dynamic product/lifecycle/lane facts were pushed back toward catalog metadata.

The Bondbuilder markdown still names OLAPLEX, K18, and Epres as technology examples because that helps the model understand the category. It now explicitly says not to recommend, rank, compare, show cards, or make product-specific claims unless select_products or curated metadata returned the relevant product truth. This keeps the context useful without turning it into a stale product fact sheet.

Bondbuilder boundaries: The old negative-only ending was rewritten as replacement behavior.

category.bondbuilder.v1 now ends with Boundaries And Replacement Behavior instead of Do Not. Each forbidden mapping is paired with what the model should do instead, so the guidance stays compact and constructive while product/tool/validator enforcement remains outside the markdown.

CWC/OWC placement: The old CWC/OWC technique docs were folded into base.general_advice.v1.

This keeps V0 lean and avoids a new topic package before Compare Lab proves the need. If CWC/OWC becomes a frequent routed intent, it may deserve a dedicated package later.

Source-map test lifecycle: I added coverage for every legacy markdown source while the old guidance directory remains the migration reference.

If the old guidance is later archived or deleted, this test should be retired or pointed at the archive manifest. Keeping it forever would turn a temporary migration audit into brittle maintenance.

Safety forbidden-text checks: The regression fixture avoids broad words like Produkt and Routine as forbidden terms.

Those words can appear in good safety answers as caveats, for example “keine Produkt- oder Routineempfehlung”. The fixture now checks for more concrete bad behaviors such as product cards, purchase language, or a routine plan.

Open Questions

CWC/OWC package split: Keep as base general guidance for now. Revisit only if manual testing shows the model needs more targeted loading or the guidance becomes too large/noisy.

Semantic intent self-reporting: The live regression report still shows cases where the model can classify a concrete product or routine ask as general advice and therefore avoid the tool validator.

I am leaving that as an explicit next design question instead of adding German regex intent gates in this pass. A future fix should preserve the architecture principle: model owns interpretation, code validates exposed structure and tool behavior, and manual/live evals catch remaining model-quality misses.

Trace retention: Compare Lab judgment logs may currently retain full AgentV2 traces, including model step items and injected session memory.

This is dev-only under tmp/, but it is broader than the analysis snapshot needs. If AgentV2 traces become more sensitive or durable, we should sanitize judgment persistence to store only review-relevant summaries.
