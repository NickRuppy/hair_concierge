# Plan Review — `plans/2026-07-12-analytics-loading-performance.md`

**Verdict:** Approve with revisions — the plan is exceptionally well-grounded (this is clearly a post-review revision), but it has two spec defects that would cause drift in a fresh executor, one fuzzy measurement step, and one unstated product tradeoff the owner must decide explicitly.

**Grounding summary.** All 30 referenced files exist. All named commands resolve (`npm run build`, `perf:mobile`, `test:node`, `ci:verify`). The suspicious-sounding claims all checked out: Next 16.2.4 really does write `.next/diagnostics/route-bundle-stats.json` with per-route `firstLoadUncompressedJsBytes` and chunk paths (`node_modules/next/dist/build/index.js:2609`); Meta's queue really is unbounded (`src/lib/meta-pixel.ts:154-167`) with stub-and-flush on consent grant (`:214-223`); PostHog inits as a module side effect with no consent gate (`src/providers/posthog-provider.tsx:12-19`); `$pageview` bypasses `trackAppEvent` (`:29`); both PostHog and Customer.io providers statically import `useAuth` → Supabase (`posthog-provider.tsx:6`, `customerio-provider.tsx:17`, `auth-provider.tsx:4-6`); Methodik and 404 inherit `LandingTracking` via `EditorialShell` (`src/components/editorial/editorial-shell.tsx:10`), firing a false `landing_viewed` (`src/providers/funnel-context-bootstrap.tsx:10`); legal pages are already clean (grep confirms no provider imports); `eventRoutes` disables Customer.io for `purchase_completed`/`subscription_started` (`src/lib/analytics/routes.ts`); the destination-spy pattern (`tests/analytics-tracking.test.ts:23`) and the brittle source-regex assertions (`tests/acquisition-funnel-tracking.test.ts:11-23`) both exist as described; the funnel keepalive POST with `crypto.randomUUID()` event IDs is at `src/lib/funnel/client.ts:37-47`; and `GET /api/funnel/session` is read-only (`src/lib/funnel/server.ts:38`), so Methodik's "context continuity without `landing_viewed`" claim is accurate.

**Lean shape**
- Irreducible goal: stop shipping vendor/auth JS in the first-load graph of public pages while every tracked event still arrives with the same names, payloads, and consent semantics.
- Cut or defer: nothing structural — prior review rounds already trimmed this (legal pages removed from scope, Meta left as-is, Sentry deferred, kill switch rejected with rationale). Two lean notes: (a) the Customer.io FIFO is mostly redundant — `AnalyticsBrowser.load()` already buffers calls until the CDN script arrives, so the new queue only needs to cover the consent-grant → dynamic-import-resolution gap; don't build more queue than that gap requires. (b) Phase 0 item 4's "rewrite the existing source-regex assertions" is premature in Phase 0 — the target module structure doesn't exist yet; move the rewrite into the phases that restructure those files, keep only the new runtime tests in Phase 0.
- Hard tradeoff the plan is avoiding: fast-bounce event loss (see Tradeoff 1 below).

**Prior art**
- Memory FIFO + stub-until-ready + flush: matches the Segment/fbq stub-and-flush pattern, with in-repo prior art the plan explicitly cites (`src/lib/meta-pixel.ts:154-182`) — OK.
- Bounded queue, drop with dev warning: canonical — OK.
- Revenue-event idempotency: preserved via `$insert_id` (`src/lib/analytics/destinations/posthog.ts:71`) and Meta `eventID` + sessionStorage dedupe (`src/lib/meta-pixel.ts:437-476`), and the constraints forbid touching them — OK.
- Per-destination consent gate + per-destination adapter + typed event map: all preserved — OK.
- Missing invariant: the readiness proxy omits one method the consumers actually call (Defect 1).

## Hard technical defects

1. **The PostHog proxy contract omits `reset`.** Phase 2 Task 9 enumerates "`capture`, `identify`, and `register`… only `get_session_id` may return `undefined` before readiness," but `PostHogIdentify` also calls `posthog.reset()` on logout (`src/providers/posthog-provider.tsx:52`). A verified enumeration of all client-side singleton calls is: `capture` ×3, `identify`, `register`, `reset`, `get_session_id`, `init`. As written, an executor building exactly the specified proxy leaves `reset` undefined — logout during SDK load would throw or silently skip clearing persisted identity. Fix: add `reset` to the queued-method list, FIFO-ordered relative to `identify`.
2. **`ready-check` doesn't resolve to anything.** The Review And Handoff section says "Run `ready-check` after Phase 4" — there is no such skill, script, or `.claude` reference in this repo or the user skill registry (grep of `.claude/`, `~/.claude/skills/`, `package.json` finds nothing). A subagent will stall or improvise. Fix: replace with the repo's actual finish steps (`npm run ci:verify` + `superpowers:verification-before-completion`, then `superpowers:requesting-code-review` and the Codex whole-branch review per CLAUDE.md's "Finishing a Feature Branch").
3. **Phase 0 item 5 (byte attribution) has no named method.** No bundle analyzer is installed (`@next/bundle-analyzer` absent from package.json), and `route-bundle-stats.json` gives per-route chunk paths and totals but not per-vendor attribution. "Attribute the first-load bytes to PostHog, Customer.io, Supabase/auth, Sentry…" is exactly the kind of fuzzy step an executor satisfies with a guess. Fix: name the method — e.g. map `firstLoadChunkPaths` to vendors by grepping chunk contents, or add `@next/bundle-analyzer` as a dev-only baseline tool.
4. **"Immediately after first paint" has no operational definition.** The plan forbids idle callbacks and long timeouts but never says what the loader trigger *is* (effect-on-mount? `requestAnimationFrame` after hydration? `window` `load`?). Too early re-enters the critical path via network contention; too late widens the event-loss window. One sentence specifying the mechanism removes the ambiguity.
5. *(Minor, wording)* Phase 2 Task 2 "Make destination SDK imports dynamic and single-flight" can be misread to include Meta, which has no npm SDK module (it's script-injected, and the Purchase path depends on `initMetaPixel()` succeeding synchronously with `bypassConsent: true`, `src/lib/meta-pixel.ts:451-469`). The Target Architecture section scopes Meta correctly; add "(PostHog and Customer.io only)" to the task line so the task list is self-consistent.

## Tradeoffs — owner decisions required

1. **Fast-bounce event loss vs. deferred loading.** Today PostHog inits eagerly and captures `$pageview` almost immediately; after this change, any visitor who bounces before the dynamically imported SDK is ready produces zero PostHog/Customer.io events — and that window is largest on slow mobile, the exact cohort being optimized. The plan asserts "must not change event names or counts," which deferred loading cannot fully guarantee, and Phase 4.7's volume monitoring only detects the drop after the fact. **Decision for Nick:** accept a small expected drop in PostHog `$pageview`/landing volumes as the cost of the performance win (and annotate it alongside the Methodik metric note), or expand scope with a first-party pageview beacon fallback. The plan currently promises the drop won't happen instead of deciding it.
2. **PostHog stays consent-ungated.** Decision 9 and the Rejected list explicitly preserve always-load PostHog; the memory index records this as an open TTDSG § 25 backlog item. This is a previously made, recorded decision — no action needed, but note the new single loader entry point makes a future consent gate a one-line wrap, worth a sentence in the plan so the follow-up lands in the right place.
3. **No kill switch, revert-only rollback.** Decision 12 states this explicitly with rationale (no active users). Canonical rollout shape would want a flag, but this is a documented owner choice, not an omission — I'd let it stand; re-confirm only if launch traffic starts before this ships.
4. **Methodik entry-path attribution timing shifts.** Only the POST handler resolves the pending touch and clears `FUNNEL_TOUCH_COOKIE` (`src/app/api/funnel/session/route.ts:37-53`). Removing Methodik's `landing_viewed` POST means a Methodik-first visitor's `entry_path` gets attached to their *next* milestone (e.g. `quiz_started`) instead of a fake landing event. Arguably an improvement, but it's a second metric-definition change beyond the event-count drop Decision 11 annotates. **Decision:** accept and fold into the same annotation.
5. **Uniform queue vs. thin Customer.io shim.** Since the CDP SDK self-buffers post-`load()`, the owner can choose symmetry (one queue mechanism for both destinations — simpler mental model) or minimalism (promise-chain for Customer.io only). Either is defensible; the plan should say which, so the executor doesn't pick silently.

## Smaller / nice-to-haves

- Name the e2e command in Phase 3/4: the smoke spec is Playwright (`tests/e2e-smoke.spec.ts`, `playwright.config.ts`) and is *not* covered by `npm run test:node` (which only globs `tests/*.test.ts{,x}`).
- The new proxy module must keep the `typeof window` guard — `tests/analytics-tracking.test.ts:14` imports the singleton in a Node test environment.
- Phase 1's new public-context module must not import `@/providers/posthog-provider` even transitively (module-scope `posthog.init` side effect); the plan states this in prose — worth repeating in the Phase 1 task itself since it's the easiest way to silently fail the phase gate.
- The Lighthouse script's default base URL is production `https://chaarlie.de` (`scripts/perf/mobile-lighthouse.mjs:8`); the plan already mandates explicit `LH_BASE_URL` — good, keep that mandatory.

**Bottom line**
This plan is ready after small fixes — the architecture is right, modeled on verified in-repo prior art, and nearly every codebase claim survived grounding. Before handing it to an executor: add `reset` to the proxy contract, replace `ready-check` with real finish steps, name the byte-attribution method and the "after first paint" trigger, and — most importantly — have Nick explicitly accept or mitigate the fast-bounce event-loss window instead of the plan promising unchanged event counts. None of these require re-shaping; they're targeted edits to an otherwise executable plan.
