**Verdict:** Approve with revisions — the strategy is sound and unusually well-grounded on the DB/reuse surface, but the first executable phase (Phase 1) has one architecture blocker and is over-scoped. Resolve those before the Phase 1 plan is written.

> Note: this is explicitly a *north-star strategy doc* ("Do not execute it as one large implementation task… the next artifact is a standalone Phase 1 plan"), so I reviewed strategic soundness + Phase 1 readiness, not verbatim-executability.

**Lean shape**
- **Irreducible goal:** Nick (solo) can see the state of every user-submitted-product intake job and publish a reviewed product without babysitting a Codex chat thread or hand-editing local package files.
- **Cut or defer:**
  - **Phase 1 ships 3 tables; only one is read in Phase 1.** Phase 1 (`plan:602`) adds jobs **+ artifacts + decisions**, but `product_intake_research_artifacts` has no consumer until Phase 2 and `product_intake_review_decisions` none until Phase 4 (`plan:644`). Ship `product_intake_research_jobs` only in Phase 1; add the other two in the phase that first reads them. Additive migrations are cheap, but RPCs/types/tests for tables nothing reads yet is speculative surface.
  - **Re-open the "separate Next app" decision against the no-monorepo reality** (see Blockers). The Decision Register calls this "light repo/app setup" (`plan:119`) — that premise is not grounded.
- **Hard tradeoff the plan avoids: build cost vs. submission volume.** The plan never states expected submissions/day, yet that single number is what justifies a durable 3-table job system + a second app + a bounded-concurrency runner vs. a single `jobs` table behind a thin status page. "Local files hide state" justifies *a* durable job row; it does not by itself justify the full cockpit. Put the expected v1 volume in the doc and let it size the durability.

**Prior art**
- **Background job / claim queue** — matches the canonical *at-least-once + idempotent + visibility-timeout + poison-message* shape: unique partial index for one non-terminal job/submission, optimistic claim RPC, retry+backoff, stale-lock TTL, max-attempt → `blocked` (`plan:215-220`). No existing SKIP-LOCKED/lock pattern in the repo (grep found none in `supabase/migrations`, `src`, `scripts`), so this is genuinely new and correctly given extra scrutiny. **Missing invariant:** the unique partial index needs the *non-terminal status set* enumerated explicitly (which of the 9 job statuses count as "active"?) or two active jobs can slip through. Pin it in the Phase 1 plan.
- **Idempotent publish/notification** — reuses the existing `notification_sent_at` conditional claim (`src/lib/product-intake/notifications.ts:239-242`, `.is("notification_sent_at", null)`). Real and correct.
- **Additive schema migration** — expand-only, constraint untouched in Phase 1 (`plan:603`). Verified: the plan's status list **exactly matches** the live DB constraint (`supabase/migrations/20260612130000_product_intake_submissions.sql:227-238`), and the safety rule "no `approved`/`matched_existing` without `approved_product_id`" (`plan:738`) matches `..._success_product_check` (`:239-242`). Clean.
- **Image judging** — correctly treated as new (`plan:865`), **but the plan ignores the existing hard-won cutout pipeline** (see High-confidence issues).

**Blockers** (will fail or regress as written when Phase 1 is split into an executable plan)
1. **The `apps/product-intake-review` separate Next app cannot import the planned `src/lib/product-intake/research-worker/*` libs.** This is a single Next app: `tsconfig.json:21-22` maps `@/*` → `./src/*` only, and there is **no workspace tooling** (no `workspaces` key in `package.json`, no `pnpm-workspace.yaml`/`turbo.json`/`nx.json`). The plan puts the app under `apps/` (`plan:439`, `:692`) and the shared logic under `src/lib/...` (`plan:540-548`) — across that boundary `@/` won't resolve and there's no second `package.json`/build wired up. Phase 1's "Add separate internal app shell under `apps/product-intake-review`" (`plan:602`) hits this on day one. **Fix (pick one, state it in the Phase 1 plan):** (a) gated route group in the *same* app — `src/app/(internal)/product-intake/…` behind middleware/env flag + separate deploy/preview (meets the "not a customer page" boundary without a second build), (b) stand up real workspaces + a shared package, or (c) keep extending the existing `node:http` server. The current review surface is already a 3,558-line raw `node:http` server (`scripts/product-intake/review-app.ts:1`), not Next — so "separate Next app" is a *new build/deploy target*, not a refactor. The boundary concern that drove rejecting `src/app/admin` (`plan:443`) is satisfiable with a gated route group; the second-app cost is not "light."

**Bottom line**
