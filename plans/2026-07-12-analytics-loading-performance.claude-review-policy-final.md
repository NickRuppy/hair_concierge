**Blockers** (will produce a wrong or divergent implementation as written)

1. The plan's Meta queue invariants contradict two money-path call sites it never mentions — `src/lib/meta-pixel.ts:427` and `:451`. (a) `trackMetaSubscriptionConfirmed` passes `queueWhenPending: false` and fires from a one-shot mount effect on checkout return (`src/app/welcome/checkout-return-analytics.tsx:33`), often followed by an immediate `router.replace`. Under the blanket "pageviews, standard events, and trackCustom events all queue before readiness," an implementer either preserves the opt-out (Subscribe stays silently droppable pre-readiness — note it already drops today on fresh-document checkout returns, because the child effect fires before the provider's init effect, so baseline diffing will mask it) or removes it (the sessionStorage "tracked" marker at `meta-pixel.ts:430-432` then gets set for a merely-queued event, permanently suppressing retry if the queue never flushes). (b) `trackMetaPurchaseConfirmed` self-starts the pixel via `initMetaPixel` + `bypassConsent` (`meta-pixel.ts:451,466`), directly violating "event capture must not start a vendor loader before the shared post-paint coordinator releases it; after release, each loader starts at most once." A subagent enforcing that invariant strictly would remove the self-init and let Purchase queue — degrading the highest-value event's delivery guarantee. **Fix:** add one paragraph naming both call sites and their intended posture (recommended: purchase keeps self-init as a sanctioned second release point; state explicitly whether Subscribe becomes queue-capable with marker-set-on-actual-dispatch or keeps the accepted drop). *Owner decision to state, not silently choose: does browser `Subscribe` need readiness-survival, or is server-side continuity (`src/app/api/stripe/webhook/route.ts:155`) sufficient?*

2. The PostHog flush trigger is specified twice, incompatibly. Phase 2 task 5 is imperative ("Register funnel context **before** flushing dependent buffered PostHog events") while Target Architecture is best-effort ("**may** enrich early buffered events") alongside "flush once in FIFO order." A literal reading of task 5 forces gating the flush on the `/api/funnel/session` GET (`src/lib/funnel/client.ts:19`); a literal reading of the architecture section flushes at SDK-ready without context. Both are defensible, so two subagents will build two different runtimes, and one of them fails the other's test. **Fix:** one sentence, e.g. "flush when the SDK is ready AND the bootstrap promise has settled (it always settles — `client.ts:30` catches to null); register context first when present, then flush the whole queue once in FIFO order."

An implementer hunting for a nonexistent pageview-queue gap wastes a cycle; reword to "custom events."

Phase 3's sequence check covers it behaviorally, but the file list should name it so the verification subagent actually loads it.
