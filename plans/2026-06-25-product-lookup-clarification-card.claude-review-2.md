**Verdict:** Approve with revisions — the plan is well-grounded and correctly shaped, but 3 blockers (a missing pipeline injection seam, a trust-boundary window during the status rename, and an undefined `category_mismatch` threshold) must be closed before a subagent — especially GPT‑5.4 — executes Tasks 4/6/1.

**Lean shape**
- **Irreducible goal:** When a user names a near-miss product, stop the repeated generic "Welche genaue Variante meinst du?" (hard-coded at `responses-agent.ts:2481`) and instead show selectable same-brand/same-category candidates with an add-product escape.
- **Cut or defer:**
  - **`category_mismatch` as a separate status** — defensible to keep (it reuses the card mechanism, as the plan argues), but it is the fuzziest and highest-risk piece, and its "identity strong enough" rule is *the original bug restated*. Either define the threshold concretely (test-anchored) in Task 1 or defer it. Don't ship it with a vague threshold.
  - **New conversation-state field vs. reuse** — the plan asserts a new "active resolved product context" field without saying why the existing `prior_selected_product_projections` slot (`persisted-session-state.ts:26`) can't carry a single verified product. Add one sentence of justification or reuse it.
  - **`product-card.tsx` subcomponent extraction** (file map, lines 292–294) — already hedged "do not over-refactor"; keep it strictly "only if needed."
  - **Phase split** — the card is decorative without Tasks 4+5. Consider shipping V1a (Tasks 1, 2, 3, 6 = status split + card render + kill repeated fallback) before V1b (Tasks 4, 5 = selection endpoint + continuity). That isolates the riskiest surface and gives a bisectable, independently-valuable increment.
- **Hard tradeoff the plan is avoiding:** the *injection seam* — how a verified selected product is fed back into the agent without (a) a forbidden parallel answer engine or (b) forbidden free-text re-parsing. No such seam exists today (see Blocker 1). The plan defers this to "implementation inspection," which is exactly where a subagent will go wrong.

**Prior art**
- **Persisted UI payload on the assistant message** — matches the existing `product_intake_offer` pattern exactly: type in `MessageRagContext` (`types.ts:821‑827`), streamed + persisted in `route.ts:449‑472`, built in `stream-events.ts:12‑36`, hydrated in `use-chat.ts:250‑259`. Correctly grounded and consistent. ✅
- **Structured card-action endpoint** (Task 4) — matches the canonical "idempotent authorized POST, not free-text overload" shape; plan lists auth → conversation ownership → message exists → product-ID ∈ persisted candidates → revalidate active. **Missing invariant: idempotency.** A double-click/replay re-runs the agent → duplicate assistant messages + duplicate state mutation. Add a dedupe/idempotency key.
- **Enum/status migration consumed across modules** — this is an expand→migrate→contract migration of a trust-bearing enum. **Missing invariant: no window where the boundary is loosened** (see Blocker 2). The plan offers an *optional* legacy alias; it must be mandatory or done in-task.

**Blockers** (will fail or regress as written)
1. **No pipeline seam to inject a verified product.** `PipelineParams` is `{message, conversationId, userId, requestId, productIntakeEnabled}` (`chat-pipeline.ts:124‑130`) and the entry is `runAgentV2ProductionPipeline` (`chat-pipeline.ts:657`) — there is **no input for a pre-resolved product**. Task 4's "call/reuse the normal AgentV2 chat answer machinery with verified selected-product context" has no concrete target; a subagent will almost certainly build the forbidden parallel engine or re-inject free text. *Fix:* name the seam (new pipeline param + how the agent is seeded so it skips re-lookup/re-clarify) in the plan before hand-off.
2. **Trust-boundary window during the rename.** Task 1 renames `ambiguous`, but the validator's `UNRESOLVED_PRODUCT_LOOKUP_STATUSES` still contains only `"ambiguous"` (`final-answer-validator.ts:254‑259`) and the deterministic fallback gates on `lookupStatus !== "ambiguous"` (`responses-agent.ts:2464‑2466`). Between Task 1 and Task 6, `needs_variant_selection`/`category_mismatch` would **not** be treated as unresolved, so product-specific claims leak past the guard. *Fix:* make the transitional alias mandatory until Task 6, or update the validator set + fallback gate in the same task as the rename.
3. **`category_mismatch` strength threshold is undefined.** The original bug is weak wrong-category neighbors (e.g. "Syoss Intense Keratin" mask sharing the token "Intense"). Today `meaningfulCandidates`/`meaningfulTokenOverlap` accept any single non-low-value token overlap (`product-lookup.ts:181‑211`) and return `ambiguous`. The plan's "meaningfully strong identity" is a fuzzy verb with no measurable criterion — and it is precisely what decides `category_mismatch` vs `not_found`. *Fix:* define it as a test-anchored rule (min meaningful-token overlap, or name-line match) in Task 1, or the new status reproduces the bug it's meant to fix.

**High-confidence issues** (correctness, not preference)
- **Additive state field touches four functions, not one.** The plan names only the normalizer; a new field also needs `createDefaultAgentV2ConversationState` (`persisted-session-state.ts:67`), `buildAgentV2State` (`:127`), and `summarizeAgentV2ConversationState` (`:98`) in lockstep. TS catches the interface-bound ones; `summarize` will silently omit it. Call all four out.
- **Task 4 state persistence.** The "no state mutation" skip is wired to `productIntakeOfferToSend` (`route.ts:486‑487`), not generic. The selection path needs its *own* state-persistence branch, not merely "not entering" the intake-deferred branch. Plan notes the intent (Task 4 bullet) but should specify the new branch.
- **`buildAssistantDecisionContext` is positional with 5 args already** (`stream-events.ts:12‑18`); adding a 6th positional `product_lookup_clarification` makes the `route.ts:466` call site fragile. Convert to an options object as part of this change.

**Smaller / nice-to-haves**
- The card payload must be built from the pipeline's `ProductLookupExecution.result` (`chat-pipeline.ts:119‑122, 180‑199`), which carries full candidates — **not** from `context.productLookupResults`, because `summarizeProductLookupResult` (`responses-agent.ts:1861‑1897`) deliberately drops candidates. Make this explicit so a subagent doesn't try to read candidates off the validator context (they won't be there).
- `none_action.product_intake_offer` should reuse the existing `buildIntakeOffer` (`product-lookup.ts:163‑179`) rather than constructing a second offer shape.
- The candidate same-category/mismatch split is already available for free: matcher tags candidates `fuzzy_candidates_review` (same category) vs `text_category_mismatch_review` (mismatch) via `reviewCandidatesByCategory` (`product-matching.ts:229, 458‑548`). This confirms the plan's "no production edit to product-matching.ts" claim. ✅ Map the payload `reason` enum off `candidate.reasonCodes`.
- File map's compare mock `run-agent-v2.ts:755` returns a lookup stub — confirm it emits a valid new status or `agent-v2-contracts.spec.ts` will drift.
- German: the `category_mismatch` sample copy uses a "Maske/Kur" badge — I found no single canonical `category_label_de` source; verify the labeler exists before the payload depends on it.

**Bottom line**
Ship after fixing the three blockers and re-shaping Task 4. The plan is grounded — file paths, the persistence pattern, the matcher's category tagging, the validator trust boundary, and all cited tests/scripts check out — and the framing (explicit status split + structured card mirroring `product_intake_offer`) is right. But as written it hands a subagent a Task 4 with no real injection seam, a rename window that silently loosens the unresolved-product guard, and a `category_mismatch` rule vague enough to reintroduce the original bug. Define the pipeline seam, make the status alias mandatory until the validator catches up, and pin the candidate-strength threshold with tests; then it's safe for hand-off. Given the size, split Task 4 into 4a (seam + server-side revalidation) and 4b (endpoint + streaming), and consider V1a/V1b phasing so the decorative card never ships ahead of the click-through.
