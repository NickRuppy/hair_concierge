**Blockers / high-confidence issues** (will drift or produce a wrong integration as written)

1. **Meta event-ID contract contradicts the code and the plan's own constraint.** The Field Name Contract says "Meta always receives the stable event ID needed for deduplication" (line 339) and Task 8 says "Verify Meta receives stable event IDs." 2. **Two public-route gates + non-boundary matching in the middleware.** Task 6 says add `/lp` to "the public marketing route classification" and `/api/funnel` to "the public and fast API route classifications." Grounding shows **two** separate structures: the early-return block `isPublicMarketingRoute` + `fastPublicRoutes` (`src/lib/supabase/middleware.ts:50-65`, this is the one that skips `supabase.auth.getUser()`) **and** the later `publicRoutes` array (`:99-129`, which only prevents the redirect). A subagent that updates only one gets a subtle bug (auth-lookup latency on every ad click, or a redirect leak). Also, the existing lists use plain `.startsWith(route)` (`:63`, `:129`) — **not** boundary-safe — so the plan's instruction to "use `pathMatchesRoutePrefix`" requires refactoring those `.some(...)` calls, not just appending a string. Call out both gates and the matching-style mismatch with these line numbers. Mirror `tests/paypal-middleware-public.test.ts` for the verification (it already asserts `updateSession(new NextRequest(...))` → 200, no `location`).

3. **The funnel `Set-Cookie` must survive `updateSession`'s redirect responses.** `src/proxy.ts:12` returns `updateSession(request)` verbatim, and for authenticated `/quiz` that return value is a fresh redirect built by `redirectWithSupabaseCookies` (`middleware.ts:277-284`). The plan says "Update `src/proxy.ts`" and "apply funnel logic for `/`, `/lp`, `/quiz`" but never specifies *where* relative to `updateSession` so the cookie attaches to the **final** response. Add an explicit step: wrap `updateSession`'s returned response and set the funnel cookie on it (covering redirect and non-redirect paths).
