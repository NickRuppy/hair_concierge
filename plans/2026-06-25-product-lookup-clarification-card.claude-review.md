**Verdict:** Approve with revisions — the shape is right, but two load-bearing gaps (an incomplete file map for the status rename, and an enum change that the compiler will not police) will cause a silent trust-boundary regression if a subagent executes the plan verbatim. Fix Blockers 1–2 before hand-off.

**Lean shape**
- **Irreducible goal:** When a user names a concrete product that is a near-miss (same brand/category, no exact row), show a structured candidate card with product-ID-based selection + an "add my product" path, instead of looping the generic "Welche genaue Variante?" clarification.
- **Cut or defer:**
  - **`image_url` on candidates / "image if available"** — the lookup catalog is loaded with `select("id, name, brand_id, product_line_id, category_key, is_active, lifecycle_status, is_chaarlie_recommended")` (`src/lib/product-intake/repository.ts:101`) and `ProductIntakeCatalogProduct` has **no image field** (`src/lib/product-intake/product-matching.ts:12-28`). There is no data source for the image. Either delete it for V1 (name + category badge is enough) or add a step to extend the catalog select + projection. As written it's an unsourced field.
  - **`category_mismatch` status** — the only grounded evidence (Syoss Intense Volume) is a same-category variant → `needs_variant_selection`. `category_mismatch` is a second, more speculative UX surface. Defer it; ship `needs_variant_selection` first. This halves the rename surface and the copy/badge work.
  - **`product-matching.ts` changes** — the plan hedges "if lookup cannot distinguish same-category vs category-mismatch." It already can: the lookup has each candidate's `product.categoryKey` vs the requested `category`, so the `reason` can be derived inside `product-lookup.ts` without touching the matcher. Likely deletable.
- **Hard tradeoff the plan is avoiding:** *How does a verified `selected_product_id` produce an answer without a re-lookup?* The lookup is entirely model-driven via the `lookup_product_candidate` tool (`src/lib/agent-v2/production/chat-pipeline.ts:784-804`); there is no mechanism to inject a pre-resolved product into the agent turn. Task 4 ("Run AgentV2/pipeline with verified selected product identity rather than reparsing text") is one checkbox over the single hardest piece of plumbing in the plan.

**Prior art**
- **Did-you-mean / candidate disambiguation:** matches the canonical shape — server-authoritative candidate IDs, client posts an opaque ID, server revalidates the ID against the original candidate set rather than parsing free text (plan "Candidate Selection Semantics" + Task 4 revalidation). OK.
- **Persisted UI-action payload on the message:** matches the existing `product_intake_offer` precedent exactly (`MessageRagContext` at `src/lib/types.ts:821-827`, built in `src/lib/chat-runtime/stream-events.ts:12-36`, streamed + persisted in `src/app/api/chat/route.ts:449-472`, patched client-side in `src/hooks/use-chat.ts:250-259`). Good reuse.
- **Status enum consumed by string-equality across modules:** *missing invariant.* `ProductLookupStatus` is a bare string union compared with `===` in multiple files. The canonical safe pattern for "add a variant to an enum that drives behavior across consumers" is an exhaustive `switch` + `assertNever`, so the compiler flags every consumer that must change. The plan adds members to an open union compared by string — TypeScript will **not** catch a missed consumer. See Blocker 2.

**Blockers** (will fail or regress as written)

1. **The Target File Map omits the files where the `ambiguous` → `needs_variant_selection`/`category_mismatch` rename must actually land.** The deterministic repair fallback that the plan's own Problem Evidence blames lives in `src/lib/agent-v2/runtime/responses-agent.ts:2455-2469` (`buildProductLookupClarificationFallback`, gated on `ambiguous`/`insufficient_identity`/`unsupported_category`), with further status branches at `responses-agent.ts:2477-2540`. The validator trust boundary the plan wants to *preserve* (Task 6) is `UNRESOLVED_PRODUCT_LOOKUP_STATUSES` at `src/lib/agent-v2/validation/final-answer-validator.ts:254-259`. The model's own behavioral contract enumerates the statuses at `src/lib/agent-v2/tools/tool-definitions.ts:125` and `responses-agent.ts:1053-1055`. **None of these four files are in the file map** — yet Task 6 and the status-mapping table depend on all of them. *Fix:* add `responses-agent.ts`, `final-answer-validator.ts`, and `tool-definitions.ts` to the map with explicit instructions for each (fallback gating, the unresolved-status set, the tool description copy, the model guidance copy). Also `src/lib/agent-v2/compare/run-agent-v2.ts:756` uses `insufficient_identity` (compare harness — lower priority but should be noted).

2. **The status rename is not compile-checked, so a partial edit silently breaks the trust boundary.** Every consumer compares the status with `===` (`final-answer-validator.ts:255-258`, `responses-agent.ts:2464-2540`, `chat-pipeline.ts:186`, `final-answer-validator.ts:1527`). If a subagent adds `needs_variant_selection`/`category_mismatch` to `product-lookup.ts` but misses `UNRESOLVED_PRODUCT_LOOKUP_STATUSES`, the validator will **stop treating those results as unresolved** and may permit product-specific claims after an unresolved lookup — the exact regression Task 6 says to prevent — with **no type error**. *Fix:* require the new statuses be added to `UNRESOLVED_PRODUCT_LOOKUP_STATUSES` in the same task, and add an explicit "grep every `=== "ambiguous"` / unresolved-set consumer" acceptance step. Consider an exhaustive `switch (status)` with `assertNever` at the fallback site so future drift is compiler-caught.

**High-confidence issues** (correctness, not preference)

- **Candidate selection cannot mutate conversation state through the current route branch.** When an offer/payload is pending, the route deliberately **skips** state mutation: `route.ts:486-501` (`product_intake_deferred_no_state_mutation`). But Task 5 requires the *selection* turn to **write** the resolved-product context. The plan must specify that the selection turn takes a different path (not the deferred branch), or the resolved-product write will be dropped. Not addressed.
- **"Active resolved product context" is genuinely new persisted state, and the existing 'named product context' is not it.** `AgentV2ConversationStateV2` (`src/lib/agent-v2/production/persisted-session-state.ts:21-29`) holds only `routine_thread_context`, `prior_selected_product_projections`, `session_memory`. `named-product-context.ts` is a **per-message regex extractor**, not persisted state (`buildAgentV2NamedProductContext`, `src/lib/agent-v2/named-product-context.ts:225-285`). So Task 5 must add a field to the versioned state + its normalizer (`normalizeAgentV2ConversationState`) + the transition builder. Additive fields are tolerated by the normalizer (it defaults missing keys), so no version bump is needed — but the plan should say so and name the normalizer, or a reviewer will flag a migration concern.
- **Category enum mismatch between the two context systems.** `named_product_context.category` is `AgentV2CareCategory`, which includes `peeling` (`named-product-context.ts:65-69`; selectable set at `persisted-session-state.ts:64`). The clarification `query.category` is `ProductIntakeCategoryKey`, which has 8 keys and **no `peeling`** (`src/lib/types.ts:324-332`). The resolved-product continuity store and the clarification payload speak different category vocabularies; reconcile or the types won't line up.
- **No structured-action input exists today.** `chatMessageSchema` is `{ message, conversation_id }` only (`src/lib/validators/index.ts:137-140`). Task 4 needs either a schema extension or the narrow endpoint the plan flags as an open question — fine to leave open, but note the client (`use-chat.ts`) currently only ever POSTs `{ message, conversation_id }` (`use-chat.ts:156-163`), so the "structured candidate selection action from the UI" is also net-new hook plumbing, not just a card.

**Bottom line**
