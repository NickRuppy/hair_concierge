**Blockers** (will fail or regress as written)

1. **`purchase_link_status` made a required zod field breaks existing tests + admin API.** Plan Task 1 Step 2 (plan:169) adds `purchase_link_status: z.enum(["available","unavailable"])` with no `.optional()`/`.default()`. `productSchema` (`src/lib/validators/index.ts:147-169`) validates admin form bodies (`src/app/api/admin/products/route.ts:238`, `src/app/api/admin/products/[id]/route.ts:92`), and `tests/admin-product-support-specs.test.ts:6-22` `buildBaseProduct` omits the field while ~7 tests assert `parsed.success === true`. A required enum fails all of them and breaks product create/edit. **Fix:** make it `z.enum([...]).nullable().optional()` (consistent with the two timestamp fields the plan *already* made optional at plan:170-172 — this is an internal inconsistency).

2. **`purchase_link_status` made a required `Product` field breaks `npm run typecheck`.** Plan Task 7 Step 1 (plan:1296) adds a non-optional field to `Product` (`src/lib/types.ts:242-271`). `tsconfig.json:25-33` includes tests via `**/*.ts(x)` and excludes only `scripts/`, so `tsc --noEmit` checks every fixture typed `: Product` — the plan updates none of them: `tests/user-memory.spec.ts:54`, `tests/chat-debug-trace.spec.ts:353`, `tests/product-matcher.spec.ts:11`, `tests/product-card-rendering.test.tsx:8`, `tests/chat-product-mentions.test.tsx:8`, `tests/product-display-model.test.ts:17` & `:72`, `tests/agent-v2-production-chat-pipeline.spec.ts:34`. **Fix:** make the field optional on `Product` (`purchase_link_status?: "available" | "unavailable" | null`), or update all ~8 fixtures in the same task.

3. **Gliss category move leaves a stale conditioner rerank-spec row.** The migration deletes only `product_conditioner_specs` (plan:253-254), but the live DB shows Gliss (`5dc2fae3…`) has **1 row in `product_conditioner_specs` AND 1 in `product_conditioner_rerank_specs`**, and the recommendation engine reads the latter (`src/lib/recommendation-engine/selection.ts:1863`). After the move, an orphaned conditioner rerank row remains and the post-apply validation (plan Step 3, plan:398-426) joins only `product_leave_in_fit_specs`, so it goes undetected. **Fix:** also `DELETE FROM public.product_conditioner_rerank_specs WHERE product_id='5dc2fae3-a0ca-4e6c-9c30-02dd192772f0';` and add an assertion that 0 conditioner spec rows remain for that id.

**High-confidence issues** (correctness, not preference)

- **The NOT-NULL contract is coupled to an unreliable scraper.** `SET NOT NULL` (plan:371-372) requires all 237 rows classified in one migration, sourced from `checkStoredLinkBuyability`. Raw `fetch().text()` against dm/Rossmann/Douglas (JS SPAs, bot-protected) usually won't contain `"in den warenkorb"` in server HTML → the `return "unavailable"` fallthrough mislabels buyable products, which the drawer then renders as `Shop-Link aktuell nicht verfügbar` (plan:1344). Combined with unbounded concurrency/no timeout (plan:1202), the run is also likely to be rate-limited mid-catalog. Recommend keeping the column nullable and treating "couldn't determine" as unknown, never as a UI-visible "unavailable".
- **`scripts/` are excluded from typecheck** (`tsconfig.json:33`). Task 3 Step 8 and Task 6 imply `npm run typecheck` validates `scripts/ingest-products.ts` and `scripts/audit-product-metadata.ts` (plan:698-703, 1244-1250) — it does not; only `eslint .` covers them. State this so the implementer doesn't assume type safety on the script edits.
- **Ingest Step 6 condition is conceptually muddled** (plan:665-674). `product.leave_in_specs` feeds the *fit* table (`product_leave_in_fit_specs` via `inferLeaveInSpecs`, `scripts/ingest-products.ts:567-573`), while the delete targets the *legacy* table `product_leave_in_specs` (`scripts/ingest-products.ts:577-587`). There is **no write path to `product_leave_in_specs`** in the script — only the delete — so "Only delete/write … when true" overstates it; when true it still only deletes with nothing to repopulate. The preserve-by-default outcome is correct, but rename it to reflect that it's purely a delete-guard, and confirm `selection.ts:1350,2108` tolerate leave-ins that now have fit specs but empty legacy specs.
